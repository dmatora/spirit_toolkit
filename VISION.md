## **Документ-видение: SpiritToolkit**

### 1. Общая цель

**SpiritToolkit** — это унифицированная платформа, состоящая из нативных мобильных приложений (iOS/Android) и веб-портала (`hub`). Цель — предоставить стабильный, удобный и современный инструмент для духовной практики, объединив функционал старых приложений `prayer` и `hourlyprayer` в единой кодовой базе.

Важно подчеркнуть, что SpiritToolkit разрабатывается для личного, некоммерческого использования единственным пользователем и не планируется к распространению. Это определяет ключевые архитектурные решения, исключая необходимость в социальных функциях, многопользовательских режимах или сложной серверной инфраструктуре.

### 2. Ключевые принципы проектирования

*   **Ориентация на действие:** UI должен минимизировать количество кликов до начала основной задачи (молитвы).
*   **Стабильность и надежность:** Приложение не должно терять состояние пользователя или создавать ложные записи в журнале из-за сбоев.
*   **Единый опыт:** Ключевой функционал должен быть доступен и консистентен на всех платформах.
*   **Стилизация компонентов:** Визуальные части интерфейса оформляются через `styled-components`; избегаем глобальных стилей и CSS-классов.
*   **Масштабируемость:** Архитектура на основе монорепозитория Nx должна позволять легко добавлять новые, независимые функциональные модули (например, "Церковный календарь", "Библия", "Dashboard").

---

### 3. Архитектура: Приложения-оболочки и Функциональные модули

Платформа состоит из двух типов сущностей:

*   **Приложения (`apps`):** Тонкие оболочки для конкретных платформ (`mobile`, `hub`). Они отвечают за конфигурацию сборки, платформо-специфичный код и рендеринг функциональных модулей.
*   **Функциональные модули (`libs`):** Переиспользуемые библиотеки, содержащие UI, логику и состояние для конкретной функции ("Молитвослов", "Dashboard").

### 4. Модуль "Молитвослов" (`prayer-feature`)

#### 4.1. Концепция

Основной модуль для ежедневной молитвенной практики. UI спроектирован по принципу "mobile-first" и написан на React Native для переиспользования на всех платформах.

#### 4.2. Структура UI и UX

**4.2.1. Основная навигация: Нижняя панель вкладок (BottomTabNavigator)**

Это основной способ навигации по ключевым разделам модуля.

*   **Вкладки:**
  1.  **Главная (Home):** Стартовый экран и основной хаб.
  2.  **Журнал (Journal):** История молитвенной практики.
  3.  **Настройки (Settings):** Управление уведомлениями и другими параметрами.

**4.2.2. Экран "Главная" (Home Screen)**

Это первое, что видит пользователь. Экран спроектирован для максимально быстрого начала молитвы.

*   **Компоненты (сверху вниз):**
  1.  **Информационная карточка "Продолжить / Праздник":**
    *   **Логика:**
      *   **Если сессия была прервана во время чтения молитвы:** Карточка показывает кнопку "Продолжить чтение: [Название молитвы]".
      *   **В ином случае:** Карточка показывает информацию "До праздника [Название] осталось X дней".
  2.  **Блок быстрого доступа (Quick Access):** Две большие кнопки: "Утреннее правило" и "Вечернее правило".
  3.  **Кнопка "Все молитвы":** Открывает боковое меню (Drawer) с полным списком остальных молитв.

**4.2.3. Экран чтения молитвы (Prayer Screen)**

*   **UI:** Простой экран, где основной контент отображается через `react-native-render-html` из локальных HTML-файлов.
*   **Логика:** Создает запись в журнале при открытии (кроме случая "Продолжить чтение") и сохраняет прогресс в `AsyncStorage`.

**4.2.4. Экран "Журнал" (Journal Screen)**

*   **UI:** Хронологический список записей с возможностью удаления.

**4.2.5. Экран "Настройки" (Settings Screen)**

*   **UI:** Управление ежечасными уведомлениями (частота, вкл/выкл).

### 5. Приложение `hub` (Web)

#### 5.1. Концепция

Веб-портал `hub` выполняет две ключевые роли. Во-первых, он является **резервной копией** основного iOS-приложения, обеспечивая доступ к ключевому функционалу "Молитвослова" на случай, если нативное приложение временно недоступно (например, при смене устройства или в ожидании обновления). В этой роли он не предназначен для параллельного использования с мобильным приложением, поэтому **не требует функций бесшовного перехода** или синхронизации состояния в реальном времени. Во-вторых, `hub` служит платформой для веб-модулей, которые не требуют тесной интеграции с мобильным опытом, например, для дашборда мониторинга личной активности (аналогично фитнес-трекеру).

#### 5.2. Структура UI и UX

**5.2.1. Техническая основа: Next.js и React Native for Web**

`hub` — это полноценное Next.js приложение, которое выступает в роли контейнера.
*   Для рендеринга "mobile-first" модулей (как `prayer-feature`) используется `react-native-web`.
*   Для рендеринга "desktop-first" модулей (как будущий `dashboard-feature`) могут использоваться стандартные веб-технологии (React).

**5.2.2. UI `hub`**

*   **Layout и Навигация:** `Sidebar` в `hub` является прямым аналогом `DrawerNavigator` из мобильного приложения. Его основная задача — навигация *внутри* модуля «Молитвослов», предоставляя прямые ссылки на страницы конкретных молитв (например, `/molitvoslov/liturgy`, `/molitvoslov/evening`). `hub` реализует **собственную навигационную оболочку (`Sidebar`)**, которая функционально **зеркальна** мобильному `DrawerNavigator`. Для этого список молитв вынесен в виде общего массива данных из `prayer-feature`, чтобы и `hub`, и мобильное приложение могли использовать его для построения своих меню.

* **UX:** Пользователь выбирает «Молитвослов» в сайдбаре, после чего видит веб-компоновку, где в основной области отображаются конкретные экраны из библиотеки. Переходы между экранами осуществляются средствами маршрутизации Next.js (например, `/molitvoslov`, `/molitvoslov/[prayerId]`).

* **Технические оговорки:**
  * **Абстракция хранилища:** Логика работы с БД должна быть абстрагирована. Модули используют единый интерфейс, который под капотом работает с `react-native-sqlite-storage` на нативе и с `IndexedDB`/`localStorage` в вебе.
  * **Платформо-зависимый код:** Функционал, недоступный в вебе (например, нативные push-уведомления), должен быть либо скрыт, либо заменен на веб-аналог, с проверкой `Platform.OS`.
  * **Навигация не является общей:** Мобильное приложение использует навигаторы (`BottomTabNavigator`, `DrawerNavigator`) из `@spirit/prayer-feature`, тогда как `hub` реализует **собственную навигационную оболочку** (маршруты Next.js, сайдбар) и рендерит **только отдельные экраны** из той же библиотеки.

### 6. Пошаговый план реализации (обновлено)

1. **Фаза 1: Базовая архитектура и разделение ответственности.**
   * Настроить монорепозиторий Nx: приложения-оболочки `apps/Prayer` (mobile) и `apps/hub` (web), плюс библиотека `libs/prayer-feature`.
   * В `libs/prayer-feature` выделить **экраны** (`HomeScreen`, `PrayerScreen`, `JournalScreen`, `SettingsScreen`), **компоненты** (рендерер текста, карта службы и т.д.), **хуки** и утилиты, чтобы они были RN Web-совместимы.
   * **Mobile (`apps/Prayer`)** остаётся тонкой оболочкой, которая рендерит **корневой навигационный компонент** (`PrayerRootNavigator`, `BottomTabNavigator`, `DrawerNavigator`) из `@spirit/prayer-feature`.
   * **Web (`apps/hub`)** **не** рендерит корневой навигатор. Вместо этого `hub` создаёт собственную веб-компоновку (Sidebar + Main) и **встраивает отдельные экраны** из `@spirit/prayer-feature`.

2.  **Фаза 2: Веб-маршруты и компоновка `hub`**
   *   Создать общий список молитв в `@spirit/prayer-feature`.
   *   Реализовать `Sidebar` в `hub`, который динамически строит навигацию на основе этого списка.

3. **Фаза 3: Данные и персистентность.**
   * Создать абстракцию хранилища с единым интерфейсом.
   * Реализовать реализации: для мобайла (SQLite/AsyncStorage), для веба (IndexedDB/localStorage). Подключить к журналу и функции «Продолжить чтение».

4. **Фаза 4: Доменные утилиты и UX.**
   * Подключить расчёт церковных праздников и связанные карточки (например, "Ближайший праздник").
   * Адаптировать визуальные компоненты и доступность для веба и мобайла.

5. **Фаза 5: Уведомления и полировка.**
   * Экран «Настройки»: на мобайле — интеграция с нативными уведомлениями (с защитой по платформе), на вебе — без нативных пушей или с веб-аналогом по возможности.
   * Единая стилизация, тестирование и финализация.

> Итог: `hub` — потребитель **экранов и UI-компонентов** из `@spirit/prayer-feature`, тогда как **навигационное дерево остаётся мобильным** (нативные навигаторы не разделяются с вебом).
